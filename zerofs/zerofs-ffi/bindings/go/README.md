# ZeroFS Go client

Go bindings for the [ZeroFS](https://github.com/Barre/ZeroFS) programmatic
client: an async, path-based client that talks 9P to a ZeroFS server, with
idiomatic Go conveniences (a range-over-func `Dir` iterator, `io.Reader` /
`io.Writer` over a `File`, and `context.Context` wrappers).

```
import zerofs "github.com/Barre/ZeroFS/zerofs/zerofs-ffi/bindings/go"
```

The package name is `zerofs_ffi`; import it with an alias (`zerofs` above) for
readable call sites.

For an HA pair, pass both nodes as one comma-separated target set. The client
finds the serving leader and re-probes both nodes after a disconnect:

```go
fs, err := zerofs.ClientConnect("node-a:5564,node-b:5564")
if err != nil {
	log.Fatal(err)
}
defer fs.Close()
```

## How this is packaged

These are cgo bindings generated by
[`uniffi-bindgen-go`](https://github.com/NordSecurity/uniffi-bindgen-go) over a
Rust `cdylib`. The Go source (`zerofs_ffi.go`, the C header `zerofs_ffi.h`, and
the hand-written `facade.go`) is committed into this directory at each release,
so `go get` pulls a complete, compilable module; you do **not** need the uniffi
generator.

What the module does **not** carry is the native library. The bindings link
against a platform-specific `cdylib` that you build from the ZeroFS Rust
sources:

| OS / arch          | library file          |
| ------------------ | --------------------- |
| Linux x86_64       | `libzerofs_ffi.so`    |
| Linux aarch64      | `libzerofs_ffi.so`    |
| macOS x86_64       | `libzerofs_ffi.dylib` |
| macOS arm64        | `libzerofs_ffi.dylib` |

Linux and macOS only: the `cdylib` uses unix-only APIs (unix-domain sockets and
unix `OsStr`), so it does not build on Windows.

There is no prebuilt binary to download: committing per-platform binaries into
the Rust repo is not acceptable, and Go has no package registry that carries
native libraries. You build the `cdylib` once for your platform and point cgo at
it (below). That step needs a Rust toolchain and a C toolchain.

The version of this module always equals the client-family version
(`zerofs-ffi/Cargo.toml::version`); the two are released in lockstep, so build
the library from the matching source tag.

## Installing

1. Add the module:

   ```sh
   go get github.com/Barre/ZeroFS/zerofs/zerofs-ffi/bindings/go@vX.Y.Z
   ```

2. Build the `cdylib` from the ZeroFS sources at the matching version (the
   `client-vX.Y.Z` tag is the same commit as the module's `vX.Y.Z` tag):

   ```sh
   git clone --depth 1 --branch client-vX.Y.Z https://github.com/Barre/ZeroFS
   cargo build --release -p zerofs-ffi --manifest-path ZeroFS/zerofs/Cargo.toml
   # -> ZeroFS/zerofs/target/release/libzerofs_ffi.{so,dylib}
   ```

3. The header `zerofs_ffi.h` ships inside the module directory, so cgo finds it
   through `CGO_CFLAGS`; the library is your `cargo build` output, found through
   `CGO_LDFLAGS`. Wire both up as below.

## Building and linking

cgo needs the header (compile time) and the library (link time). The header is
in the module; the library is your build output. They live in different
directories, so point `-I` and `-L` at each:

```sh
export CGO_ENABLED=1
MODDIR="$(go list -m -f '{{.Dir}}' github.com/Barre/ZeroFS/zerofs/zerofs-ffi/bindings/go)"
LIBDIR=/path/to/ZeroFS/zerofs/target/release    # where cargo put the cdylib
export CGO_CFLAGS="-I$MODDIR"
export CGO_LDFLAGS="-L$LIBDIR -lzerofs_ffi"

go build ./...
```

`-lzerofs_ffi` resolves `libzerofs_ffi.so` / `libzerofs_ffi.dylib` by the
platform's normal naming rule, so the same `-l` flag works on Linux and macOS.

## Running

The `cdylib` is loaded dynamically at runtime, so the OS loader must find it:

```sh
# Linux
export LD_LIBRARY_PATH=$LIBDIR:$LD_LIBRARY_PATH
# macOS
export DYLD_LIBRARY_PATH=$LIBDIR:$DYLD_LIBRARY_PATH
```

Alternatively, bake an rpath into the binary at link time so no environment
variable is needed at run time:

```sh
# Linux / macOS
export CGO_LDFLAGS="-L$LIBDIR -lzerofs_ffi -Wl,-rpath,$LIBDIR"
```

## Regenerating the bindings (maintainers)

You normally never need this; the generated sources are committed at each
release. To regenerate against a local checkout:

```sh
# from the zerofs workspace root
cargo install uniffi-bindgen-go \
  --git https://github.com/NordSecurity/uniffi-bindgen-go --tag v0.7.0+v0.31.0
cargo build -p zerofs-ffi            # debug cdylib carries the uniffi metadata
./zerofs-ffi/bindings/generate.sh go out/go
# generated into out/go/zerofs_ffi/ (zerofs_ffi.go + .h + facade.go)
```

The release pipeline (`.github/workflows/publish-client.yml`, the `publish-go`
job) regenerates these files into this directory, commits them onto `main`, and
pushes the `zerofs/zerofs-ffi/bindings/go/vX.Y.Z` subdir tag. It builds no
binaries and creates no GitHub Release.
