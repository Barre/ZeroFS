# Standalone Compactor

ZeroFS can run compaction as separate worker processes on their own machines, moving compaction's CPU and I/O load off the main server and scaling it horizontally. {{ className: 'lead' }}

## Overview

Compaction is a background process in ZeroFS's LSM tree storage engine. It merges sorted runs to:

- **Reduce storage space** by removing old versions of updated/deleted data
- **Improve read performance** by reducing the number of files to search

Compaction is split into two roles:

- A **coordinator** that schedules compactions and commits their results to the manifest. It owns the compaction epoch, so there must be exactly one — ZeroFS binds it to the read-write database, so it runs only on the current leader (and follows the leader on failover).
- One or more **workers** that execute the scheduled jobs. Workers are stateless: they claim jobs from the shared object store, run them, and report back. You can run as many as you want.

By default, `zerofs run` runs the coordinator together with an embedded worker, so a single node compacts itself with no extra processes. For demanding workloads, you can disable the embedded worker and offload execution to a fleet of standalone workers.

## Architecture

<Row>
  <Col>

    To offload compaction:

    1. **Writer instance**: Run `zerofs run --no-compactor`. The coordinator keeps scheduling compactions and committing results, but runs no embedded worker — it executes nothing itself.
    2. **Worker instance(s)**: Run `zerofs compactor`. Each is a stateless worker that claims and executes scheduled jobs. Run one or several.

    All instances access the same object storage backend. The coordinator publishes work and the workers claim it through the shared object store; only the coordinator writes the manifest, preserving the single-writer invariant.

    <Note>
      With `--no-compactor`, at least one worker must be running, or scheduled compactions never execute and level-0 SSTs accumulate. The default `zerofs run` (embedded worker) has no such dependency.
    </Note>

## Usage

### Keeping Coordination on the Writer, Offloading Execution

Start your main ZeroFS server with the `--no-compactor` flag:

```bash
zerofs run -c zerofs.toml --no-compactor
```

The coordinator keeps running — it's bound to the read-write database and follows the leader on failover — but its embedded worker is disabled, so the server schedules compactions without executing any. Scheduled work waits for a standalone worker to pick it up.

### Starting Standalone Workers

On one or more separate instances (or the same machine), run a worker:

```bash
zerofs compactor -c zerofs.toml
```

Run as many as you need — they share the work by claiming jobs from the object store, so adding workers increases aggregate compaction throughput. Each worker reads the same configuration file to access the storage backend, uses the `[lsm]` settings for compaction parameters, and honors the `[storage]` `storage_class`. Workers apply the same encryption and compression as the writer, so they work correctly with encrypted volumes.

<Note>
  A worker only needs access to the object storage backend. It doesn't run any network servers (NFS, 9P, NBD).
</Note>

## Configuration

Standalone workers use the same configuration file as the main server. The relevant sections are:

```toml
[storage]
url = "s3://my-bucket/zerofs-data"
encryption_password = "your-password"

[aws]
access_key_id = "..."
secret_access_key = "..."

[lsm]
max_concurrent_compactions = 8  # Jobs a single worker runs in parallel
```

## When to Offload Compaction

Consider running standalone workers when:

- **Horizontal scaling**: Run multiple workers to push aggregate compaction throughput beyond a single node.
- **Egress costs**: Compaction reads and rewrites large amounts of data during merges. Workers in the same region/zone as the S3 bucket avoid cross-region data transfer fees; the main ZeroFS server can run anywhere.
- **High write throughput**: Compaction competes with reads and writes for CPU and I/O.
- **Latency-sensitive workloads**: Isolate compaction spikes from user requests.
- **Cost optimization**: Workers are stateless, so they can run on cheaper spot/preemptible instances and be safely interrupted.

For most workloads, the default embedded worker is sufficient and simpler to operate.
